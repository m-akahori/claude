name: sub-task
description: Notionチケットの子課題を自動生成する
argument-hint: "<NotionチケットのURL>"
allowed-tools: mcp__notion__get-page mcp__notion__get-block-children mcp__notion__query-database mcp__notion__create-page mcp__notion__update-page mcp__notion__update-block mcp__notion__append-blocks
---

引数に指定したNotionチケットの内容を読み込み、適切な子課題を生成・作成する。

使用例: `/sub-task https://www.notion.so/xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx`

---

## Step 1: 親チケットの読み込み

引数のURLからページIDを抽出し、`mcp__notion__get-page` で親チケットを取得する。

- URLの末尾32文字（ハイフンなし）がページID
- 例: `https://www.notion.so/abc123...def456` → page_id = `abc123...def456`
- UUIDフォーマット（8-4-4-4-12）でも可

取得したページから以下の情報を確認する：

| 確認項目 | プロパティ名 |
|---------|-----------|
| チケットNo | `チケットNo` |
| タイトル | ページタイトル |
| カテゴリ | `カテゴリ` |

次に `mcp__notion__get-block-children` でページ本文（対応内容・対応詳細・対応手順・チェックリスト）も取得する。

---

## Step 2: カテゴリ判定

取得した `カテゴリ` プロパティの値で処理を分岐する。

| カテゴリ値 | 処理 |
|----------|------|
| `運用・保守` | **[運用保守フロー]** へ |
| それ以外（`画面（Web UI）` / `API（JSON）` / `バッチ／スケジュール` / `非同期ジョブ（Queue）` 等） | **[開発フロー]** へ |

---

## [運用保守フロー]

### Step 3-A: 分割方法をユーザーに確認

ユーザーに以下のメッセージをそのまま送る：

> 運用・保守チケットの子課題を作成します。  
> どのようにチケットを分割するか、箇条書きでいいので教えてください。

ユーザーの回答を待つ。

### Step 4-A: 子課題候補の生成

ユーザーの入力を読み取り、親チケットの「対応詳細」「対応手順」「チェックリスト」のどの部分に対応するかを判別する。

判別結果をもとに子課題候補を以下の形式で表示し、ユーザーに確認する：

```
以下の内容で子課題を作成します。よろしいですか？

| # | タイトル（仮） | 内容の根拠 |
|---|-------------|---------|
| 1 | [親No]XXXXX | ユーザー入力の〇〇に対応 |
| 2 | [親No]YYYYY | ユーザー入力の〇〇に対応 |
...

y で作成 / n でキャンセル
```

ユーザーが `y` を入力したら **[共通: Step 5]** へ進む。  
`n` の場合は「キャンセルしました。」と表示して終了。

---

## [開発フロー]

### Step 3-B: 子課題の自動抽出

親チケットの本文（対応内容・対応詳細・対応手順）を精査し、以下のルールで子課題候補を生成する。

#### 必須生成ルール

| 条件 | 生成する子課題 | タイプ |
|------|-------------|-------|
| DB変更がある（migration / seeder / テーブル追加・変更・削除など） | DB子課題 | `DB` |
| Vue.js / Bladeファイルの変更がある | フロントエンド子課題 | `frontend` |
| Controller以下のロジック変更がある（Service / Repository / Model / ルーティング / 認証含む） | バックエンド子課題 | `backend` |
| フロントエンド・バックエンド問わずロジック変更がある | テストコード子課題 | `test` |

複数条件に該当する場合は、それぞれの子課題をすべて生成する。

### Step 4-B: 子課題候補の確認

抽出した特徴と子課題候補をユーザーに以下の形式で提示する：

```
親チケットの分析結果：
- DB変更: あり / なし
- フロントエンド変更: あり / なし
- バックエンド変更: あり / なし
- テストコード必要: あり / なし

以下の内容で子課題を作成します。よろしいですか？

| # | タイトル（仮） | タイプ |
|---|-------------|-------|
| 1 | [親No]DB対応 - XXXXX | DB |
| 2 | [親No]フロントエンド実装 - XXXXX | frontend |
| 3 | [親No]バックエンド実装 - XXXXX | backend |
| 4 | [親No]テストコード実装 - XXXXX | test |

y で作成 / n でキャンセル
```

ユーザーが `y` を入力したら **[共通: Step 5]** へ進む。  
`n` の場合は「キャンセルしました。」と表示して終了。

---

## [共通] Step 5: チケットNo の採番

`mcp__notion__query-database` でデータベース全件を取得し、既存の「チケットNo」プロパティの最大値を確認する。

- database_id: `2e353e837a8a80bc932ce6b1e87950cb`
- filter: なし（全件取得）
- `has_more` が true の場合は `next_cursor` を使って全件取得すること

最大値 + 1 から順に、子課題の数だけ連番でチケットNoを採番する。

---

## [共通] Step 6: 子課題の作成

確認済みの子課題を1件ずつ `mcp__notion__create-page` で作成する。

### プロパティ設定

| プロパティ | 値 |
|----------|---|
| タイトル | `[親チケットNo]内容を端的に表すタイトル`（例：親がNo.12なら `[12]バックエンド実装 - ユーザー登録API`） |
| ステータス | `TO DO`（固定） |
| カテゴリ | 親チケットと同じ値 |
| 優先度 | `中`（固定） |
| チケットNo | Step 5 で採番した番号 |
| 開始日 | 親チケットの開始日と同じ |
| 終了日 | 親チケットの終了日と同じ |
| 親課題 | 親チケットのページIDをリレーションで設定 |
| タイプ（開発のみ） | 該当する値（`DB` / `frontend` / `backend` / `test`）を設定。運用保守の場合は設定しない |

### ページ本文

以下の構造で本文を作成する：

```
# 概要
親チケット「[親タイトル]」の子課題。[この子課題でやること]を1〜2文で記載。

# 対応内容

## as is
親チケットの as is を引用しつつ、この子課題に関係する現状を記載。

## to be
この子課題が完了した状態を記載。

## 対応詳細
親チケットの対応詳細のうち、この子課題に関係する部分を抜粋・整理して記載。

# 対応手順
この子課題の具体的な実装手順を記載。

# チェックリスト
この子課題が完了したと判断するためのチェックリストを記載。
- [ ] チェック項目1
- [ ] チェック項目2
...
```

---

## [共通] Step 7: 親チケットの更新

すべての子課題作成後、親チケットを以下の4点について更新する。

### 7-1. 子課題プロパティの更新

`mcp__notion__update-page` で親チケットの `子課題` プロパティに、作成した全子課題のページIDをリレーションで追加する。

### 7-2. 本文内のリンク設定

`mcp__notion__get-block-children` で親チケットの本文を再取得し、以下を確認する：

- 「対応詳細」「対応手順」の中に、作成した子課題と明確に対応する記述がある場合、該当テキストに子課題へのリンクを設定する
- 対応が不明瞭な場合はリンクを設定しない（誤リンクを避ける）

### 7-3. 対応詳細・対応手順の整合性修正

子課題の作成によって親チケットの「対応詳細」「対応手順」に以下のような修正が必要な場合は `mcp__notion__update-block` で更新する：

- 子課題に移管した作業内容は「→ 子課題 [子チケットNo] で対応」と注記する
- 順序・粒度が子課題の構成と合っていない場合は調整する

### 7-4. チェックリストの整合性修正

親チケットのチェックリストが子課題の構成と合っていない場合、以下のように修正する：

- 子課題ごとに「- [ ] [子チケットNo] の完了」という項目を追加（または既存項目を置き換え）
- 子課題で対応する細かい手順チェックは親チケットから削除し、子課題側に移す

---

```
子課題を作成し、親チケットを更新しました。

親チケット: No.[親チケットNo] [親タイトル]

作成した子課題:
- No.[チケットNo1] [タイトル1]
- No.[チケットNo2] [タイトル2]
...
```
