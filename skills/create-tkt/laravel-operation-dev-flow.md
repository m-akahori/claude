# Laravel 運用・保守 決定フローチャート

対象：既存機能の変更・削除、コード最適化、バグ修正

```
START
│
▼
【1. 作業種別】
├─ 既存機能の変更
├─ 機能削除
├─ コード最適化（リファクタリング）
└─ バグ修正
│
▼
【2. 影響範囲の調査】※全種別共通
├─ 変更対象：ルート / コントローラー / モデル / ビュー / ジョブ
├─ DB変更が伴うか？（マイグレーション要否）
├─ 他機能・他モジュールへの依存関係
└─ 外部（APIクライアント・連携サービス）への影響
│
▼
【種別ごとの分岐】
│
├─ ■ 既存機能の変更
│   ├─ 後方互換性を壊すか？
│   │   └─ YES → バージョニング / 段階的移行 / 利用側への周知
│   ├─ DB変更を伴うか？
│   │   ├─ マイグレーション設計（down() も書く）
│   │   └─ データ移行が必要か？（seeder / 変換スクリプト）
│   ├─ 認証・認可の変更が必要か？
│   ├─ 既存テストの修正
│   └─ 新たなテストケースの追加
│
├─ ■ 機能削除
│   ├─ 使用箇所の全洗い出し
│   │   └─ routes / controllers / views / jobs / events / tests
│   ├─ DBテーブル・カラムの削除可否
│   │   └─ 他テーブルからの参照（FK）確認
│   ├─ 段階的削除か即時削除か
│   │   └─ 段階的 → まず非表示・アクセス不可 → 一定期間後にコード削除
│   └─ マイグレーション（テーブル / カラム削除）
│
├─ ■ コード最適化（リファクタリング）
│   ├─ パフォーマンス問題か？
│   │   ├─ N+1問題 → Eager Loading（with()）
│   │   ├─ 重いクエリ → インデックス追加 / クエリ見直し
│   │   └─ 繰り返し集計 → キャッシュ（Cache::remember）
│   ├─ 可読性・保守性の問題か？
│   │   ├─ Controllerが肥大化 → Service層 / Action層に切り出し
│   │   ├─ 重複コード → trait / 共通メソッド化
│   │   └─ 命名が不明瞭 → リネーム（影響範囲を全検索してから）
│   ├─ 振る舞いを変えないことを確認（テスト必須）
│   └─ リファクタリング前後でテストをグリーンに保つ
│
└─ ■ バグ修正
    ├─ 再現手順の確認
    ├─ 原因特定（ログ / テスト / dd・dump）
    ├─ 修正範囲の最小化（他機能を巻き込まない）
    ├─ 修正後に再現しないことを確認
    └─ 再発防止のテストを追加
│
▼
【3. リリース前チェック】※全種別共通
├─ テストがすべてグリーンか
├─ マイグレーションの down() は書いたか（ロールバック可能か）
├─ 本番DBへの影響（ロック時間 / データ量 / 実行時間）
└─ デプロイ手順（マイグレーション実行タイミング等）
│
▼
END
```

---

## 判断早見表

| 状況 | 確認すること |
|------|------------|
| カラム削除したい | FK参照なし？他モデルで $fillable / $casts に残ってないか |
| クエリが遅い | `->toSql()` でクエリ確認 → EXPLAIN → インデックス or Eager Loading |
| Controller が長い | 1メソッド20行超えたら Service 切り出しを検討 |
| 機能を消したい | `git grep` で全使用箇所を洗い出してから着手 |
| バグの原因不明 | まずテストで再現させてから修正する |
