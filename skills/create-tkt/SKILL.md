Notionデータベース（ID: `2e353e837a8a80bc932ce6b1e87950cb`）にチケットを新規作成する。

フローチャートの詳細は同ディレクトリの参照ファイルを使用する：
- 新規開発: [laravel-feature-dev-flow.md](laravel-feature-dev-flow.md)
- 運用・保守: [laravel-operation-dev-flow.md](laravel-operation-dev-flow.md)

---

## Step 1: 目標ヒアリング

以下の質問をユーザーに送る（そのまま）：

> あなたが成し遂げたいことは何ですか？箇条書きでいいので回答してください

ユーザーの回答を待つ。

---

## Step 2: 第一判断 — 大カテゴリの決定

ユーザーの回答内容から以下のどちらかを判定する。

| 判定 | 判断基準 |
|------|---------|
| **新規開発** | 新しい機能・画面・API・処理を追加・作成する内容 |
| **運用・保守** | 既存機能の変更・削除・最適化・バグ修正の内容 |

判定が明確でない場合はユーザーに確認する。

---

## Step 3: 小カテゴリの決定

### 「新規開発」の場合

以下のいずれかを判定（または質問して確認）する：

1. 画面（Web UI）
2. API（JSON）
3. バッチ／スケジュール
4. 非同期ジョブ（Queue）
5. その他

### 「運用・保守」の場合

以下のいずれかを判定（または質問して確認）する：

1. 既存機能の変更
2. 機能削除
3. コード最適化（リファクタリング）
4. バグ修正
5. その他

---

## Step 4: フローチャートに従い詳細を確認

小カテゴリが決まったら、対応するフローチャートを読み込み、必要事項をユーザーに質問して確認する。

### 新規開発の場合 → [laravel-feature-dev-flow.md](laravel-feature-dev-flow.md) を参照

フローチャートに基づき、以下の観点でユーザーに確認が必要な項目を質問する：

**【DB仕様】（全種別共通）**
- 新規テーブルが必要か？
- 既存テーブルを変更するか？
- リレーションの有無

**【認証・認可】（全種別共通）**
- 認証が必要か？
- 認可（Policy/Gate）が必要か？

**【種別固有の項目】**
- 画面（Web UI）: ルーティング、コントローラー種別、フロントエンド技術、レスポンス形式
- API（JSON）: エンドポイント、認証方式、リクエスト/レスポンス仕様
- バッチ／スケジュール: 実行頻度、べき等性、エラー処理方針
- 非同期ジョブ: トリガー、キュー設定、リトライ方針

**【テスト方針】**
- Feature Test / Unit Test の要否
- カバレッジ範囲（正常系のみ？異常系・境界値まで？）

### 運用・保守の場合 → [laravel-operation-dev-flow.md](laravel-operation-dev-flow.md) を参照

フローチャートに基づき、以下の観点でユーザーに確認が必要な項目を質問する：

**【影響範囲の調査】（全種別共通）**
- 変更対象（ルート/コントローラー/モデル/ビュー/ジョブ）
- DB変更が伴うか？
- 他機能・外部への影響

**【種別固有の項目】**
- 既存機能の変更: 後方互換性の有無、DB変更の有無、認証・認可変更の有無
- 機能削除: 使用箇所の洗い出し、DB削除可否、段階的削除か即時削除か
- コード最適化: パフォーマンス問題か保守性問題か、振る舞いを変えないことの確認
- バグ修正: 再現手順、原因特定結果、修正範囲

**【リリース前チェック項目の確認】**
- テスト有無、マイグレーションのdown()要否、本番DBへの影響

確認が必要な項目はまとめて質問し、ユーザーの回答を待つ。全ての必要情報が揃ったら Step 5 へ進む。

---

## Step 5: チケットNo の採番

`mcp__notion__query-database` でデータベース全件を取得し、既存の「チケットNo」プロパティの最大値を確認する。

- database_id: `2e353e837a8a80bc932ce6b1e87950cb`
- filter: なし（全件取得）
- has_more が true の場合は next_cursor を使って全件取得すること

チケットNoの形式を確認し、最大値 + 1 を新しいチケットNoとする。

---

## Step 6: チケット作成

収集した情報をもとに `mcp__notion__create-page` でチケットを作成する。

### プロパティ設定

| プロパティ | 値 |
|----------|---|
| タイトル | 内容を端的に表す適切なタイトル（日本語）|
| ステータス | `TO DO`（固定） |
| カテゴリ | 新規開発: 小カテゴリ名 / 運用・保守: `運用・保守` |
| 優先度 | `中`（固定） |
| チケットNo | Step 5 で採番した番号 |
| 開始日 | 作業開始の見込み日（仮）を適切に判断して設定 |
| 終了日 | 作業完了の見込み日（仮）を適切に判断して設定 |

### ページ本文（概要欄）

以下のMarkdown構造でページ本文を作成する：

```
# 概要
対応種別（例：新規開発 - 画面（Web UI））と、このチケットでやることを1〜3文で簡潔に記載する。

# 対応内容

## as is
現状を記載する。新規開発の場合は「該当機能なし」など。

## to be
あるべき姿・このチケットのゴールを記載する。

## 対応詳細
Step 4 で収集した種別固有の情報（DB仕様、認証・認可、エンドポイント等）をここに記載する。

# 対応手順
概要と対応内容を達成するための具体的な手順を詳細に記載する。

# チェックリスト
このチケットが完了したと判断するためのチェックリストを記載する。
- [ ] チェック項目1
- [ ] チェック項目2
...
```

## 完了メッセージ

チケット作成完了後、以下の形式で結果を表示する：

```
チケットを作成しました。

No: [チケットNo]
タイトル: [タイトル]
カテゴリ: [カテゴリ]
開始日: [開始日] 〜 終了日: [終了日]
```
